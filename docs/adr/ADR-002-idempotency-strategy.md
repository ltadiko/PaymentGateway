# ADR-002: Idempotency Strategy

**Status:** Accepted  
**Date:** 2026-03-10  
**Deciders:** Senior Development Team

---

## Context

Payment processing is a critical financial operation. In distributed systems, clients may retry requests due to:

- Network timeouts (client never received the response, but the server processed it).
- Load balancer retries.
- Client-side retry logic with exponential backoff.

Without idempotency protection, a single payment intent could result in **duplicate charges** — an unacceptable outcome for a payment gateway.

---

## Decision

### Client-Provided Idempotency Key

- The client **must** include an `Idempotency-Key` HTTP header on every `POST /api/v1/payments` request.
- The key must be a UUID v4, generated by the client for each unique payment intent.
- Missing or malformed keys result in `400 Bad Request`.

### Database-Backed Idempotency Store

- An `idempotency_keys` table stores:
  ```
  | Column           | Type         | Constraint                          |
  |------------------|--------------|-------------------------------------|
  | id               | UUID (PK)    | Auto-generated                      |
  | tenant_id        | VARCHAR      | NOT NULL                            |
  | idempotency_key  | VARCHAR      | NOT NULL                            |
  | payment_id       | UUID         | FK → payments.id                    |
  | response_status  | INT          | HTTP status code of original response|
  | response_body    | TEXT (JSON)  | Serialized original response        |
  | created_at       | TIMESTAMP    | NOT NULL                            |
  |                  |              | UNIQUE(tenant_id, idempotency_key)  |
  ```

- The unique constraint is on `(tenant_id, idempotency_key)`, meaning different tenants can independently use the same key value without collision.

### Processing Flow

```
Client sends POST /api/v1/payments with Idempotency-Key header
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ Check: Does (tenant_id, idempotency_key) exist?     │
└──────────────────────┬──────────────────────────────┘
                       │
              ┌────────┴────────┐
              │                 │
           EXISTS           NOT FOUND
              │                 │
              ▼                 ▼
    Return original      ┌─────────────────────────┐
    stored response      │ BEGIN TRANSACTION        │
    (HTTP 200)           │  1. Insert IdempotencyKey│
                         │  2. Create Payment       │
                         │  3. Save AuditTrail      │
                         │ COMMIT                   │
                         └──────────┬──────────────┘
                                    │
                                    ▼
                           Publish Kafka event
                           Return HTTP 202 Accepted
```

### Race Condition Handling

If two identical requests arrive simultaneously:

1. Both pass the initial `EXISTS` check (both see "NOT FOUND").
2. Both attempt to insert into the `idempotency_keys` table.
3. The **database unique constraint** causes one insert to fail with a `DataIntegrityViolationException`.
4. The failing request catches this exception, re-queries the existing record, and returns the original stored response.

This guarantees **exactly-once semantics** at the database level without requiring distributed locks.

### Key Expiration

- Idempotency keys are retained for **24 hours** (configurable).
- A scheduled task (`@Scheduled`) purges expired keys.
- This prevents unbounded table growth while still protecting against delayed retries.

---

## Alternatives Considered

| Alternative | Why Rejected |
|---|---|
| **Redis-based idempotency cache** | Adds an external dependency (Redis). For this assignment, a DB-backed approach is simpler and equally correct. In production, Redis with TTL would offer better performance for high-throughput scenarios. |
| **Server-generated idempotency key** | Shifts the deduplication burden to the server, which cannot distinguish a retry from a new request. Client-provided keys are the industry standard (Stripe, PayPal, Adyen all use this pattern). |
| **Hash-based deduplication** (hash the request body) | Fragile — semantically identical requests with different field ordering or whitespace produce different hashes. Also, two legitimately different payments could have identical amounts/recipients. |
| **Optimistic locking only** | Prevents concurrent modification but does not prevent duplicate creation — a second request would create a new payment, not return the existing one. |

---

## Consequences

### Positive
- Industry-standard approach, well-understood by API consumers.
- Database-level guarantee prevents duplicates even under high concurrency.
- Original response is stored and returned verbatim on retries — clients get consistent behavior.
- Tenant-scoped keys prevent cross-tenant interference.

### Negative
- Requires clients to generate and manage idempotency keys (minor burden).
- Stored responses consume additional database space (mitigated by TTL-based cleanup).
- The `response_body` column stores serialized JSON — schema evolution requires care.

### Compliance Note
- The 24-hour retention window aligns with typical payment retry windows. For regulatory purposes, the **payment** and **audit trail** records are retained indefinitely — only the idempotency key mapping is purged.

## Production Caching Considerations

### Does caching help prevent duplicate payments?

**No — caching alone cannot prevent duplicates. But it improves performance.**

| Layer | Role | Prevents Duplicates? |
|---|---|---|
| **Cache (Redis)** | Fast-path duplicate detection (~1ms) | ❌ Not authoritative — two requests can miss simultaneously |
| **DB unique constraint** | Ultimate guard — one INSERT succeeds, the other throws | ✅ Yes — only reliable mechanism |
| **Idempotency key lookup** | Returns stored response for known keys | ⚡ Performance optimization |

### Why Caffeine (local cache) was NOT implemented

Caffeine is an in-process cache. In a distributed payment gateway with multiple instances behind a load balancer:

```
Instance A: Client sends payment → Caffeine caches idempotency key ✓
Instance B: Same client retries  → Caffeine has NO idea about it ✗ → hits DB anyway
```

Local caching only helps if the same client always hits the same instance (sticky sessions), which is not guaranteed and should not be assumed.

### Recommended Production Approach: Redis

For high-throughput production deployments, use **Redis** as a distributed cache with a three-tier lookup strategy:

```
1. Check Redis cache (~1ms)         ← fast-path for known duplicates
   │
   ├── HIT  → return stored response (no DB query)
   │
   └── MISS → 2. Check database (~5-10ms)  ← authoritative lookup
                  │
                  ├── HIT  → populate Redis, return stored response
                  │
                  └── MISS → 3. Create payment + idempotency key
                                  │
                                  └── DB unique constraint = ultimate guard
                                      On success → write to Redis + DB
                                      On DataIntegrityViolationException → re-query DB
```

**Redis configuration:**
- **TTL:** Match idempotency key retention (e.g., 24 hours)
- **Key format:** `idempotency:{tenantId}:{idempotencyKey}`
- **Value:** Serialized response JSON
- **Eviction:** LRU when max memory reached (safe — DB is the source of truth)

This was intentionally left out of the assignment implementation because:
1. Adding Redis increases infrastructure complexity
2. The assignment explicitly says "use embedded databases" and avoid complex infrastructure
3. The DB-only approach is **correct** — Redis is a performance optimization, not a correctness requirement
